Cotato 프로젝트를 진행하면서 사용자에게 메일을 보내야 할 때가 있다. 
예를 들면 '메일 인증번호 전달' 이나 '권한 변경 안내 메일' 등이 있다. 
이 때 빠른 개발을 위해 JavaMailSender를 이용했다. 
하지만 새로운 기능에는 더 안정적이고 모니터링이 가능한 Amazon SES를 이용하기로 결정하였다. 
이번 포스트에서는 '모집 시작 메일 신청 + 전송 기능' 개발 기록을 작성하겠다.



## 기능 설명
동아리에서는 6개월에 한번씩 새로운 회원 모집을 지원한다. 
기존에 동아리에 흥미를 갖고 있는 인원들은 다음 모집 기수가 언제 시작하는지 알 수 있는 방법이 수시로 홍보 게시물을 확인하는 방법 외에 없다. 
이 때문에 모집이 시작될 때 메일로 알려주는 기능이 있으면 좋겠다는 결론에 도달하였다.


핵심 기능은 다음과 같다.

1. 사용자가 모집 시작시 메일로 전송을 할 것(이하 모집 알람 메일)을 신청할 수 있다.

2. 관리자가 모집 알람 메일을 신청한 인원들에게 메일을 전송할 수 있다.

3. 모집 알람 메일 전송 로그를 확인할 수 있다.


## ERD 설계

'모집 메일 신청자'는 신청한 모든 사용자의 정보를 담고 있다.

이 때 send_status는 다음과 같다.
```
    NOT_SENT("안보냄")
    FAIL("전송 실패")
    SUCCESS("전송 성공")
```


관리자가 모집 알람 메일 신청자에게 메일을 전송할 때 모집_메일_전송_요청 tuple을 하나 생성한다.

이후 각 로그를 모집_메일_전송_로그 테이블에 저장하는 방식이다.



## 기능 1. 사용자가 모집 알람 메일을 신청할 수 있다
사용자가 전달 받을 메일을 입력하면 서버는 '해당 메일이 이미 신청됐는지 확인'한다.

이 때 사용자는 여러 기수에 신청할 수 있기 때문에 아직 보내지 않은 상태(NOT_SENT)일 때만 예외를 발생한다.



이후 모집_메일_신청자 테이블에 해당 내용을 저장한다.


```java
@Transactional
public void requestRecruitmentNotification(String recruitEmail) {
    if (recruitmentNotificationRequesterReader.existsByEmailAndSendStatus(recruitEmail, SendStatus.NOT_SENT)) {
        throw new AppException(ErrorCode.ALREADY_REQUEST_NOTIFICATION);
    }

    recruitmentNotificationRequesterRepository.save(
            RecruitmentNotificationRequester.of(recruitEmail, isPolicyChecked)
    );
}
```


## 기능 2. 관리자가 모집 알람 메일을 신청한 인원들에게 메일을 전송할 수 있다
관리자가 모집 알림 메일을 신청한 사용자들에게 메일을 전송하는 기능은 다음과 같은 5단계로 구성된다.



1. 보내야 하는 메일 목록 반환
2. 모집_메일_전송_요청 튜플 생성
3. 비동기로 메일 전송
4. 메일 전송 결과(send_status) 업데이트
5. 로그 저장



### 1. 메일 전송 대상자 목록 조회
먼저, 메일을 전송해야 하는 사용자 목록을 조회한다. 
이 목록은 이전에 메일 전송이 실패했거나 아직 전송되지 않은 사용자를 대상으로 조회한다.

```java
// Service 계층
List<RecruitmentNotificationRequester> allNotSentOrFailRequester =
    recruitmentNotificationRequesterReader.findAllNotSentOrFailEmails();

// RecruitmentNotificationRequesterReader 클래스
public List<RecruitmentNotificationRequester> findAllNotSentOrFailEmails() {
    return recruitmentNotificationRequesterRepository.findAllBySendStatusIn(
        List.of(SendStatus.NOT_SENT, SendStatus.FAIL));
}
```


### 2. RecruitmentNotification 객체 생성
메일 발송 작업을 추적하기 위해, 관리자의 정보와 모집 기수를 담아 RecruitmentNotification 객체를 생성하고 저장한다.

```java
//notification 객체 생성
RecruitmentNotification recruitmentNotification = recruitmentNotificationRepository.save(
RecruitmentNotification.of(member, generationNumber));
```

이후 동적으로 메일 제목 + html body를 반환 받는다.

```java
EmailContent emailContent = recruitmentEmailFactory.getRecruitmentEmailContent(generationNumber);
//EmailContent에는 메일 제목, htmlBody가 들어있다.
```

### 3. 비동기 메일 전송
여러 사용자에게 메일을 한번에 보낼 경우 서로의 메일 주소가 노출될 위험이 있다. 
따라서, 사용자별로 개별적으로 메일을 비동기 처리하여 전송한다.

List<CompletableFuture<NotificationResult>> notificationTasks = allNotSentOrFailRequester.stream()
        .map(requester -> recruitmentNotificationSender.sendNotificationAsync(requester, emailContent))
        .toList();

비동기 메일 전송 로직(sendNotificationAsync 메소드)

```java
@Async("emailSendThreadPoolExecutor")
public CompletableFuture<NotificationResult> sendNotificationAsync(final RecruitmentNotificationRequester requester,
                                                                    final EmailContent emailContent) {
    return sendEmailAsync(requester, emailContent)
        .thenApply(v -> NotificationResult.of(requester.getId(), true))
        .exceptionally(e -> {
                log.info("메일 전송 실패 email: {} exception: {}", requester.getEmail(), e.getMessage());
                return NotificationResult.of(requester.getId(), false);
        });
}

private CompletableFuture<Void> sendEmailAsync(final RecruitmentNotificationRequester requester,
                                                final EmailContent emailContent) {
    return CompletableFuture.runAsync(() ->
            sesEmailSender.sendRawMessageBody(
                requester.getEmail(),
                emailContent.htmlBody(),
                emailContent.subject()
            ));
}
```
각각의 메일 전송 후 성공했다면 thenApply 메소드가 실행되고 실패하면 exceptionally 메소드가 실행된다. 
이렇게 한 이메일 마다 그 결과 값(NoticiationFuture) 가 저장돼 최종적으로 모든 메일의 전송 결과(List<NoticiationResult> results)를 얻을 수 있다.




### 4. 메일 전송 결과(send_status) 업데이트
   비동기로 로직을 수행할 때 메일 전송 결과를 수정하는 방식이 있는데 왜 따로 notificationResults를 관리하는지 의문이 들 수 있다.

하지만 해당 항목을 수정하기 위해서는 메일 수 만큼의 쿼리가 나가게 된다. 
이로 인해 성능에 문제가 발생할 수 있기 때문에 한번에 처리해야 한다.

```java
private void updateRequesterSendResult(List<NotificationResult> results) {
        List<Long> successIds = results.stream()
            .filter(NotificationResult::success)
            .map(NotificationResult::requestId)
            .toList();
        List<Long> failIds = results.stream()
            .filter(r -> !r.success())
            .map(NotificationResult::requestId)
            .toList();

	if (!successIds.isEmpty()) {
		recruitmentNotificationRequesterRepository.updateSendStatusByIds(SendStatus.SUCCESS, successIds);
	}
	if (!failIds.isEmpty()) {
		recruitmentNotificationRequesterRepository.updateSendStatusByIds(SendStatus.FAIL, failIds);
	}
}
```



### 5. 메일 전송 로그 저장
각 전송 결과를 기반으로 log 튜플 리스트를 만든다. 
이 또한 대량의 로그를 JPA로 저장하는 것이 아닌 Jdbc batch를 이용해 저장해 빠른 속도로 저장할 수 있도록 했다.


```java
private void saveEmailLogs(List<RecruitmentNotificationRequester> requesters,
                           List<NotificationResult> results, 
                           RecruitmentNotification notification) {
    Map<Long, RecruitmentNotificationRequester> requesterById = requesters.stream()
            .collect(Collectors.toMap(RecruitmentNotificationRequester::getId, Function.identity()));

	List<RecruitmentNotificationEmailLog> logs = results.stream()
		.map(notificationResult -> RecruitmentNotificationEmailLog.of(
        	requesterById.get(notificationResult.requestId()),
			notification,
			notificationResult.success()
		))
		.toList();
    recruitmentNotificationEmailLogJdbcRepository.saveAllWithBatch(logs);
}
```

## 기능 3. 모집 알람 메일 전송 로그를 확인할 수 있다.
전체 코드

```java
@Transactional(readOnly = true)
public RecruitmentNotificationLogsResponse findNotificationLogs() {
List<RecruitmentNotification> top5Notification = recruitmentNotificationReader.findTop5LatestNotifications();

	Map<Long, List<RecruitmentNotificationEmailLog>> logsByNotificationId = recruitmentNotificationEmailLogReader.groupByNotificationIds(
		top5Notification);

	List<RecruitmentNotificationLogResponse> responses = top5Notification.stream()
		.map(notification -> RecruitmentNotificationLogResponse.of(
			notification,
			logsByNotificationId.getOrDefault(notification.getId(), List.of())
		))
		.toList();
	return RecruitmentNotificationLogsResponse.of(responses);
}
```

먼저 최근 5개의 모집 알람 요청 튜플을 가져온다.


각 요청들의 로그를 한번의 쿼리로 가져온 후 요청에 따른 기록으로 그룹핑한다.

```java
public Map<Long, List<RecruitmentNotificationEmailLog>> groupByNotificationIds(
        List<RecruitmentNotification> notifications) {
    List<Long> notificationIds = notifications.stream()
        .map(RecruitmentNotification::getId)
        .toList();
    return recruitmentNotificationEmailLogRepository.findAllByNotificationIdIn(notificationIds).stream()
		.collect(Collectors.groupingBy(log -> log.getNotification().getId()));
}
```

이후 요청을 응답 Response로 반환한다.
