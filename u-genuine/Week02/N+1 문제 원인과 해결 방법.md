ORM을 쓰다 보면 흔히 마주하는 이슈 중 하나가 바로 N+1 문제다.

ORM은 객체와 데이터베이스 간의 변환을 자동으로 처리해주기 때문에 편리하지만, 

의도치 않은 성능 문제가 발생하기도 한다.

---

### **N+1 문제란?**

한 번의 쿼리를 실행한 뒤, 결과로 가져온 N개의 엔티티 각각에 대해 추가 쿼리를 실행하면서 총 N+1번의 쿼리가 발생하는 현상이다.

예를 들어 게시글을 한 번에 100개를 조회했는데, 각 게시글마다 댓글을 따로 조회하게 되면 쿼리는 총 101번 실행된다.

이렇게 연관된 데이터를 하나씩 조회할 때 발생하는 반복 쿼리 문제가 N+1이다.

---

### **연관 관계를 명시하지 않아도 발생한다?**

N+1 문제는 @ManyToOne이나 @OneToMany같은 연관 관계에서 Lazy Loding 때문에 발생하는 것만은 아니다.

현재 진행 중인 프로젝트에서는 JPA 연관 관계 어노테이션 없이

매핑 엔티티를 별도로 두고 ID 값만으로 관계를 맺고 있다.

그런데 이런 방식에서도 N+1 문제는 발생한다.

---

### **현재 프로젝트 도메인 구조 예시**

- **게시판(Board):** 여러 개의 게시글을 포함

- **게시글(Post):** 특정 게시판에 소속

- **카테고리(Category):** 게시글에 태그처럼 붙는 분류

- **게시글-카테고리 매핑(PostCategory):** 게시글 카테고리를 연결하는 다대다 매핑 테이블

#### **엔티티 구성 예시**

```
// Post.java
Long id;
Long boardId;
String content;
...

// category.java
Long id,
String categoryName;

// PostCategory.java
Long id;
Long postId,
Long categoryId
```

---

### **어떤 상황에서 N+1 문제가 생겼을까?**

특정 게시판에서 특정 카테고리에 해당하는 게시글만 조회하는 기능이 필요했고

당시에는 다음과 같은 방식으로 구현을 했다.

1. 게시판 ID로 게시글 목록 조회

2. 각 게시글이 PostCategory를 조회해, 카테고리 포함 여부 확인

#### **코드 예시 (문제의 코드)**

```
// 1. boardId로 전체 게시글 조회
List<Post> posts = postRepository.findByBoardId(boardId);

List<Post> filteredPosts = new ArrayList<>();

for(Post post : posts) {
	// 2. 각 게시글마다 PostCategory 조회
    List<PostCategory> postCategories = postCategoryRepository.findByPostId(post.getId());
    
    boolean hasTargetCategory = postCategories.stream()
    	.anyMatch(postCategory -> postCategory.getCategoryId().equals(targetCategoryId);
    
    if (hasTargetCategory) {
	    filteredPosts.add(post);
    }
}
```

**이 로직의 문제**

게시글이 100개라면 postCategoryRepository.findByPostId(...)가 100번 실행됨

- 게시글 조회 쿼리 1번

- 게시글마다 PostCategory 조회 쿼리 100번 -> 총 101번의 쿼리

N+1 문제 발생!

---

### **해결 방법 1. IN 쿼리로 PostCategory 한 번에 조회하기**


반복문 안에서 쿼리를 매번 실행하지 않고,

게시글 ID들을 모아서 IN 조건으로 PostCategory를 한 번만 조회하면 된다.

```
// 1. boardId로 전체 게시글 조회
List<Post> posts = postRepository.findByBoardId(boardId);

List<Long> postIds = posts.stream()
	.map(Post::getId)
    .collect(Collectors.toList());

// 2. 조회된 모든 게시글 ID들로 한번에 PostCategory 조회
List<PostCategory> postCategories = postCategoryRepository.findByPostIdIn(postIds);

Set<Long> filteredPostIds = postCategories.stream()
	.filter(postCategory -> postCategory.getCategoryId().equals(targetCategoryId))
    .map(PostCategory::getPostId)
    .collect(Collectors.toSet());
    
List<Post> filteredPosts = posts.stream()
	.filter(post -> filteredPostIds.contains(post.getId())
    .toList();
```

**이렇게 하면**

- 게시글 조회 쿼리 1번

- PostCategory IN 조회 1번 -> 총 2번의 쿼리

쿼리 수가 N+1 -> 2로 줄어들게 된다.

---
### **해결 방법 2. JPQL로 join 쿼리 작성**



```
SELECT p FROM Post p
JOIN PostCategory pc ON p.id = pc.postId
WHERE p.boardId = :boardId
AND pc.categoryId = :categoryId
```


### 마무리
- N+1 문제는 초기에는 잘 감지되지 않지만, 데이터가 많아지면 쿼리 수가 폭증해서 성능 저하로 이어질 수 있다.

- 연관 관계를 명시하지 않고 ID 기반으로 연동하는 프로젝트에서도

실제 쿼리 작성 방식에 따라 언제든 N+1은 발생할 수 있다는 점을 기억해두면 좋을 것 같다 :)
