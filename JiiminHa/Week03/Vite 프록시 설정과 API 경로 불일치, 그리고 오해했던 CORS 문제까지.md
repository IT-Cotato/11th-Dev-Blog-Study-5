Cotato 프로젝트에서 날씨 API 기반 로그인 기능을 구현하며, 프론트와 백엔드가 실제로 연결되는 순간을 처음 경험하게 되었다.
그 과정에서 단순한 경로 설정 문제 하나가 어떻게 예상치 못한 500 에러로까지 이어질 수 있는지를 온몸으로 겪었다.
처음으로 백엔드와 협업하면서 내가 겪은 문제와 배움을 정리해보려 한다.

### 🧭 처음엔 단순한 구조라고 생각했다

프론트엔드에서 API 요청 경로를 /api/v1/...처럼 깔끔하게 관리하고 싶었다.
그래서 Axios 요청을 다음과 같이 구성했고

```ts
axios.post("/api/v1/users/login", data, {
  withCredentials: true,
});
```

Vite의 개발 서버에서 이 요청이 백엔드로 잘 전달되도록 하기 위해 다음과 같이 프록시 설정을 추가했다.

```ts
// vite.config.ts
proxy: {
  '/api': {
    target: 'http://localhost:5173',
    changeOrigin: true,
    rewrite: path => path.replace(/^\/api/, '') // ← 여기서 잘라냈다
  }
}
```

이 설정은 /api/v1/... 요청을 백엔드에는 /v1/...으로 전달하는 역할을 한다.
나는 백엔드가 /api 접두어 없이 API를 구성했을 거라고 생각해 별다른 논의 없이 rewrite를 적용했다.

### 🧨 하지만 백엔드는 달랐다

백엔드에는 /v1/users/login이라는 경로로 요청이 들어갔고,
백엔드는 /api/v1/... 경로를 기대하고 있었기에
내가 rewrite로 잘라낸 요청은 매핑 실패 → 500 에러로 이어졌다.

에러 로그에는 이런 메시지가 반복되었다:
Unexpected error: No static resource v1/users/login. at /v1/users/login

### 🌀 혼란의 한가운데 – 내가 의심한 건 CORS였다

이 상황에서 브라우저 개발자 도구를 확인해보니, 쿠키가 저장되지 않았고,
나는 자연스럽게 CORS 문제를 의심하게 되었다.

프론트는 withCredentials: true 옵션으로 요청을 보내고 있었고,
응답에 Access-Control-Allow-Credentials: true가 포함되지 않으면 브라우저는 쿠키를 저장하지 않는다.

그래서 백엔드에 다음과 같은 메시지를 보냈다:

> 혹시 CORS 관련 문제일까요?
> 현재 프론트에서 로그인 요청을 보낼 때 withCredentials: true 옵션을 사용하고 있는데요,
> 응답 헤더에 Access-Control-Allow-Credentials: true가 포함되지 않아 브라우저가 Set-Cookie를 차단하고 있는 상황입니다.
> 이로 인해 세션이 저장되지 않아 인증이 실패하게 돼요.

> 혹시 아래 CORS 관련 헤더 값을 응답에 추가해주실 수 있을까요?
> 추가로, 로그인 시 Set-Cookie가 제대로 내려오는지도 확인 부탁드립니다!
> 프론트에선 요청은 잘 보내고 있고, 응답도 오지만 500 에러가 납니다.
> Swagger에선 잘 동작하고 있어서, 세션 처리 쪽 문제일 수도 있을 것 같아요.

### 🔍 진짜 원인은 ‘rewrite’

하지만 문제의 본질은 쿠키도, CORS도 아니었다.
로그를 꼼꼼히 확인한 끝에, 백엔드에 전달된 요청 경로가 /v1/...으로 잘못 들어간다는 점을 발견했고,
그 원인이 Vite 설정의 rewrite였다는 걸 알게 되었다.

결국 나는 아래와 같이 설정을 수정했다:

```ts
// 잘못된 설정
rewrite: (path) => path.replace(/^\/api/, '')

// 수정 후: rewrite 제거
// 프론트에서 /api/v1/...로 요청하면 백엔드에도 동일하게 전달되도록
이후 요청은 /api/v1/users/login 그대로 백엔드에 전달되었고,
매핑이 정상적으로 일어나면서 500 에러도, 쿠키 문제도 사라졌다.
```

### 🧠 회고: 500 에러는 ‘백엔드의 책임’일까?

이번 경험을 통해 정말 많이 배웠다.

500 에러는 무조건 백엔드의 문제라고 단정할 수 없다.
프론트의 요청 경로 설정(rewrite) 하나가 잘못되면, 백엔드는 아무 것도 못하고 500을 낼 수 있다.
withCredentials, CORS 헤더, 세션 처리처럼 겉으로는 관련 있어 보이는 요소들도 실제로는 문제와 무관할 수 있다.
그래서 더더욱 중요한 건, 확신 없이 추측하지 않고, 로그와 흐름을 하나하나 확인하는 태도였다.
백엔드와 협업할 땐, 사소한 설정 하나도 반드시 맞춰야 하며, 독단적인 판단은 협업 장애가 될 수 있다.

✅ 마무리

“그냥 경로 하나 잘못 넣은 거예요”
결론만 보면 사소하지만, 이 문제를 파악하고 해결하기까지 거친 모든 과정은 나에게 엄청난 실전 학습이었다.

앞으로도 협업 과정에서 에러가 발생하면,
내가 보낸 요청이 정말 백엔드가 기대한 그대로였는지 먼저 점검해볼 것이다.
